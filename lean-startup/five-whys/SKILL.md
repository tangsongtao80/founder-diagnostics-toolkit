# Five Whys（五个为什么）

来自《精益创业》埃里克·莱斯。Five Whys 是一种简单但强大的根本原因分析技术：遇到问题时连续追问五次「为什么」，直到找到系统层面的根因而非人的错误。

## 核心理念

> "大多数表面上是人的错误，实际上是系统的错误——Five Whys 帮你找到那个系统缺陷。"

莱斯强调：Five Whys 不是为了追责，而是为了建立一个每次出问题都会变得更强的组织学习系统。

## 适用场景

- 产品出了线上事故
- 用户投诉或流失激增
- 关键项目延期
- 同样的错误反复出现
- 团队复盘时想避免「下次注意」这种无效结论

## 执行步骤

### 步骤 1: 描述问题
用事实而非判断描述发生了什么。
- ✅ "用户注册后 30% 没有完成邮箱验证"
- ❌ "用户太懒不验证邮箱"

### 步骤 2: 问第一个「为什么」
回答必须用客观事实。

### 步骤 3: 继续追问
每一轮追问建立在上一轮的回答之上，直到达到系统层面（通常 3-7 轮）。如果答案开始循环，说明需要从不同角度重新切入。

### 步骤 4: 识别根本原因
当答案指向「流程缺陷」「制度缺失」「知识缺口」等系统问题时，说明找到了根因。

### 步骤 5: 按比例投入修复
这是莱斯特有的观点：根据问题的严重程度决定修复投入——小问题做小修复，大问题做大修复。避免过度工程。

### 步骤 6: 记录并共享
将分析过程和修复措施记录为组织知识，避免团队其他人在别处踩同样的坑。

## 输出格式

```
## 问题描述
（客观事实描述）

## Five Whys 分析链
### 第一层：为什么会发生？
- 原因：

### 第二层：为什么会[上一层原因]？
- 原因：

### 第三层：为什么会[上一层原因]？
- 原因：

### 第四层：为什么会[上一层原因]？
- 原因：

### 第五层：为什么会[上一层原因]？
- 原因：

## 根本原因
（系统层面的根因总结）

## 按比例修复
| 修复措施 | 负责 | 截止日期 | 投入级别（小/中/大）|
|:---|:---|:---|:---|
| | | | |

## 组织学习
- 如何避免类似问题在其他地方发生：
- 需要更新的流程/文档/工具：
```

## 注意事项

- 不要把「人的错误」当根本原因——大部分操作失误背后都有流程或培训问题
- 按比例修复：5 分钟的问题用 5 分钟修复，不要用 5 天的重构
- 让直接当事人参与分析，但避免变成「批斗会」
- 如果 Five Whys 给出了相互矛盾的原因，可能有两个根本原因同时起作用
